This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Development workflow

Every feature or change follows this workflow â€” keep it in mind from the first line of code:

**When in doubt, ask.** Any uncertainty about *what* to build â€” the requirement, the desired behaviour, the functional scope â€” must be resolved by asking the user before implementing. Never guess at intent. *Technical* decisions (architecture, patterns, data structures, naming, test design, refactors), by contrast, are made autonomously to the highest programming standards â€” do not ask about those, just apply best practice. In short: ask about the functional/feature side, decide the engineering side yourself.

1. **Plan deliberately.** Design so the software stays *extensible* (new cases slot in without rewrites) and *well testable* (logic separable from Android/UI, behaviour reachable by a test). Prefer a reusable building block over a one-off when the same need will recur.
2. **Implement** the plan.
3. **Remove dead code.** Delete now-unused sections, wrappers, imports, and helpers so the codebase stays a clear picture of what's actually in use â€” no leftovers from the previous shape of the code.
4. **Validate** the change against the requirement (and edge cases) by reading it back.
5. **Write tests** covering the new behaviour (ViewModel logic â†’ JVM unit tests under `app/src/test/`; UI behaviour â†’ Compose tests under `app/src/androidTest/`).
6. **Compile and test from the command line** using the commands below â€” `assembleDebug` + `testDebugUnitTest` (and `connectedDebugAndroidTest` for instrumented tests when a device/emulator is attached). Don't consider a step done until it builds and tests pass.
7. **Update the docs** â€” keep this `CLAUDE.md` in sync with new architecture, design decisions, and conventions.
8. **Commit** with a clear, conventional message.

### Tests

Unit tests live in `app/src/test/` and run on the JVM (`testDebugUnitTest` â€” no Robolectric). Conventions:

- **Fakes, no mocking library** (`.../fake/`): `FakeFoodEntryRepository`, `FakeGoalRepository`, `FakeFoodSearchRepository`, `FakeWeightRepository`, `FakeCustomFoodRepository`, `FakeSettingsRepository`, `FakeBackupExporter`/`FakeBackupImporter` (each with a `whileRunning` hook so a test can observe the transient `isLoading` state), and for the workout module `FakeExerciseCatalogRepository`, `FakeWorkoutTemplateRepository`, `FakeWorkoutSessionRepository` (the session fake encodes the "at most one active session" invariant).
- Tests use `kotlinx-coroutines-test` + `turbine`. **Every ViewModel has a test class** covering its full behaviour; keep that invariant when adding VM logic. A `WhileSubscribed` StateFlow only computes real values while collected â€” tests keep it hot with a `backgroundScope` collector and then read `.value`.
- **Pure domain/series logic gets its own test class** (`MacroCalculatorTest`, `InlineHistoryTest`, `RestTimerTest`, `WorkoutMetricsTest`, `TemplateUpdateTest`, `ExerciseHistoryTest`, `WeightSeriesTest`, `WorkoutSeriesTest`, `WorkoutFormatTest`, â€¦) â€” cover the edge cases, not just the happy path.
- Entityâ†”domain mapping is kept in pure, Android-free helpers so it can be unit-tested without Room (`WorkoutMappers` â†’ `WorkoutMappersTest`; the CSV formats â†’ `BackupImporterTest`/`WorkoutBackupTest`). Repository behaviour is tested with in-memory fake DAOs + a pass-through `TransactionRunner` (`WorkoutRepositoriesTest`).
- **Locale-sensitive formatting in tests**: `currentAppLocale()` falls back to `Locale.getDefault()` on the JVM, so a test asserting formatted decimals must pin `Locale.setDefault()` in setup and restore it afterward (see `WorkoutFormatTest`) â€” otherwise the expected separator depends on the machine running the test.
- `app/src/androidTest/` holds the **instrumented** tests (run with `connectedDebugAndroidTest` against an emulator/device): `WorkoutMigrationTest` exercises the real Room migrations v7â†’v8â†’v9â†’v10â†’v11 on Android's SQLite (creates the old DB, runs the migration, asserts new tables/columns + preserved data + FK cascade). Room's schema JSON is exported to `app/schemas/` (`exportSchema = true` + the `room.schemaLocation` KSP arg); a migration's CREATE statements are copied verbatim from the generated schema JSON so the migrated schema matches the entities exactly.

### Backup & data portability

- **Auto Backup**: `backup_rules.xml` (pre-12) and `data_extraction_rules.xml` (Android 12+) explicitly include `trainist.db` + WAL files â€” Android backs these up to Google Drive automatically, no code required.
- **Interface seam â€” `domain/backup/`.** `BackupExporter`/`BackupImporter` are **interfaces** in the domain layer, implemented by `BackupExporterImpl`/`BackupImporterImpl` in `data/backup` (bound in `RepositoryModule`), so `DataViewModel` follows the same "depends only on interfaces" rule as every other VM. The seam carries the backup location as the **string form of a `Uri`**, not a `Uri`: that keeps the interfaces (and therefore the VM) Android-free, so the whole export/import flow is reachable by a plain JVM test â€” `Uri` round-trips losslessly through `toString()`/`Uri.parse()`, the impls parse it, the UI re-parses it for `Intent.EXTRA_STREAM`, and nothing in between inspects it.
- **Full Backup Export** (`data/backup/BackupExporterImpl`): reads all data â€” nutrition (food entries, weight entries, daily goal, custom foods) and training (custom exercises, templates + slots, sessions + exercises + sets) â€” and writes a ZIP to `cacheDir` with one named CSV per section (`food_entries.csv`, `weight_entries.csv`, `daily_goal.csv`, `custom_foods.csv`, `exercises.csv`, `workout_templates.csv`, `template_exercises.csv`, `workout_sessions.csv`, `session_exercises.csv`, `set_entries.csv`), shared via `FileProvider` + `Intent.ACTION_SEND`. Only **custom** exercises are exported â€” catalog entries are reproduced by the bundled-snapshot seeding on every install and resolve by stable id.
- **Full Backup Import** (`data/backup/BackupImporterImpl`): opens a user-picked URI, auto-detects ZIP vs single CSV via the PK magic bytes. A ZIP imports every known section by entry name (unknown/missing entries ignored); a single CSV is routed by `detectCsvType` (column presence â€” see below). The Android IO is kept thin; dispatch/parse/save lives in the Android-free `importZipEntries(InputStream, ImportTargets)` / `importCsvLines(List<String>, ImportTargets)`, unit-tested end-to-end in `BackupImporterTest` (incl. legacy nutrition-only backups, a full training round-trip and a fresh-device relink).
- **Relational training data â€” stable-key reassembly** (`data/backup/WorkoutBackup.kt`, pure, `WorkoutBackupTest`): the training graph is split into flat CSVs whose **foreign keys are the stable string keys** (`Exercise.stableId`, `WorkoutTemplate.stableId`, `WorkoutSession.stableId`), **never** row ids. On import, `assembleTemplates`/`assembleSessions` re-group the rows (`set_entries` link to their parent by `(session_stable_id, exercise_position)`; rows sorted by `position`), then each root is saved fresh (`id = 0`, target mints its own row ids). Custom exercises are `upsertAll`-ed first so references resolve on a fresh device. The relational sections only reassemble as a whole ZIP; a lone `exercises.csv` imports standalone, the other five training CSVs are recognised-but-no-op on their own.
- **CSV type detection** (`detectCsvType`): identifies a CSV by a unique column. Training sections are matched **first** (`set_entries.csv` also carries `weight_kg`, so it must win over the weight-entries rule):
  - `is_custom` â†’ `EXERCISES`; `template_name` â†’ `WORKOUT_TEMPLATES`; `target_sets` â†’ `TEMPLATE_EXERCISES`; `is_active` â†’ `WORKOUT_SESSIONS`; `superset_group_id` â†’ `SESSION_EXERCISES`; `exercise_position` â†’ `SET_ENTRIES`
  - `food_name` â†’ `FOOD_ENTRIES`; `weight_kg` â†’ `WEIGHT_ENTRIES`; `kcal_per_100g` â†’ `CUSTOM_FOODS`; `kcal` (without the above) â†’ `DAILY_GOAL`
- **CSV formats** (all pure Kotlin, one object per section, name-based columns; `CsvFormat.parseHeaders` is the shared header parser): `CsvFormat` (food entries, incl. `tag` â†’ `FoodTag.parse`, missing â†’ `NONE`), `CustomFoodCsvFormat`, `WeightCsvFormat` (`date`, `weight_kg`, `timestamp_ms`), `GoalCsvFormat` (single row; optional `target_weight_kg`), `ExerciseCsvFormat`, `WorkoutTemplateCsvFormat`, `TemplateExerciseCsvFormat` (`set_types` = planned `SetType`s joined by `|`; older backups fall back to `target_sets` NORMAL sets), `WorkoutSessionCsvFormat` (optional `template_stable_id`), `SessionExerciseCsvFormat`, `SetEntryCsvFormat`. List cells (muscles/instructions/set types) are `|`-joined â€” one line per row; an embedded newline would break the line-based reader.
- **FileProvider** authority: `dev.antonlammers.trainist.fileprovider`, paths in `res/xml/file_paths.xml` (cache dir).

### Backup schema evolution â€” backward compatibility contract

All CSV parsing is name-based (column order irrelevant). This gives forward and backward compatibility for free, as long as these rules are followed:

| Change | Export (`toRow` / `HEADER`) | Import (`fromRow`) |
|---|---|---|
| **Add a column** | Add constant to the format object, add to `HEADER` and `toRow` | Read with a sensible default for missing values â€” old exports still parse |
| **Remove a column** | Remove from `HEADER` and `toRow` | Remove the read; unknown columns are ignored, so new exports still parse in old app versions |
| **Rename a column** | Write both old and new names for at least one release, then drop the old | Read the new name with fallback to the old until the transition ends |
| **Add a new data type** | New `*CsvFormat` object + ZIP entry name in `BackupExporter` | New `CsvType` variant + detection rule in `detectCsvType` (more-specific rules before any that shares a column), handle in `importZipEntries` + `importCsvLines` |
| **Add a relational data type** | Flat CSVs whose foreign keys are **stable string keys** (never row ids); one section per table | Reassemble the graph in a pure helper (like `WorkoutBackup`), save each root with `id = 0`; keep it a whole-ZIP operation |
| **Change the ZIP container format** | â€” | Extend `isZip` / add a new container detection path; keep old paths for legacy files |

**No explicit version field is needed.** Structural detection (column presence, ZIP magic bytes, entry names) is sufficient. Only introduce a version field if two formats become structurally ambiguous.

**Referential integrity across devices.** Relational backups never serialise auto-increment row ids â€” all cross-row references use stable string keys, so a backup rebuilds correctly on a device whose row ids differ.

### Key design decisions

- **Domain layer is Android-free.** ViewModels depend only on `domain/repository` interfaces. Swapping the data source touches only `data/` and `di/`.
- **Bottom navigation â€” four tabs** (`AppNavigation`, `bottomNavItems()`): **ErnĂ¤hrung** (`Screen.Overview`), **Training** (`Screen.Workout`, the template list), **Statistik** (`Screen.Stats`), **Einstellungen** (`Screen.Settings`). Icons active = `Icons.Rounded.*`, inactive = `Icons.Outlined.*` (Restaurant / FitnessCenter / BarChart / Settings). The nav is **icon-only** â€” no `label`; the localized name lives on as each icon's `contentDescription`. Goal editing lives behind the Settings hub (no Goals tab). Any route not in `bottomNavItems` (`AddFood`, `BarcodeScanner`, `ExerciseCatalog`, `ExerciseDetail`, `TemplateEditor`, `WorkoutSession`, `WorkoutHistory`, `Goals`, `DataBackup`, `Support`) hides the bar.
- **Theme isolation â€” "Ink & Paper".** Replacing `ui/theme/Color.kt`, `Theme.kt`, and `Type.kt` restyles the entire app. Fixed monochrome color system (no Material You, `dynamicColor = false`): dark = "Ink", light = "Paper", chosen from the system setting (`isSystemInDarkTheme()`), no in-app toggle. Colour is reserved for data. `Color.kt` holds the `Ink*`/`Paper*` palettes (paper/bg/surface/surface2/line/text/text2/text3), the mode-independent `AccentLilac` (`#B79FCB`, `primary`/`secondary`) + `OnAccentInk`, the pastel macro colours (`CalorieColor`/`ProteinColor`/`CarbsColor`/`FatColor`) and the clean-eating tag colours. `Theme.kt` maps these onto the Material3 schemes (`surfaceContainerLowest` = chrome, `outlineVariant` = hairline, `error` = unhealthy tone). The base XML theme `Theme.Trainist` must stay **DayNight-aware** (`values/themes.xml` light + `values-night/themes.xml` dark window background) â€” otherwise the window background bleeds through as a hairline at the bottom-nav seam in dark mode.
- **Typography â€” three downloadable Google fonts** (`ui/theme/Type.kt`): `NewsreaderSerif` (display/headline/title: screen titles, big numbers), `HankenGroteskSans` (body + `labelLarge`), `JetBrainsMono` (`labelMedium`/`labelSmall` uppercase micro-labels with wide tracking, e.g. `KCAL`). Loaded at runtime via the Google Fonts provider (`font_certs.xml`) â€” no bundled font binaries; requires Google Play Services at runtime.
- **Icons â€” Material Symbols Rounded.** All icons use `Icons.Rounded.*` (not Filled/Default). Reach for the Rounded variant when adding a new icon.
- **Numeric input â€” `ui/components/NumericTextField`.** *Every* numeric field uses it, never a bare `OutlinedTextField`. On focus it **selects its whole content** so a pre-filled value is overwritten by simply typing. String-based `value`/`onValueChange` API (callers parse, accepting comma or period via `normalizeDecimal`); `decimal = false` for whole-number keyboards; nullable `label`; optional `textStyle`/`leadingIcon`/`placeholder` slots (the placeholder shows grey inline-history hints in the session).
- **Drag-to-reorder â€” `ui/components/DragReorderColumn`.** The one reordering pattern app-wide (template editor slots, session sets, templates list, stats cards): long-press the row's drag handle (`Icons.Rounded.DragIndicator`, haptic tick), drag, settles on release. Renders a plain non-lazy `Column` (nest it as a single item inside an outer `LazyColumn`); takes `items`/`key`/`onMove(from, to)` plus an `itemContent` slot receiving a `rowModifier` (apply to the row container â€” carries drag translation/elevation) and a `dragHandleModifier` (apply only to the handle so the gesture never fights the row's own controls). While dragging it calls `onMove` **once per adjacent step** (whenever the dragged row crosses a neighbour's mid-height), so the ViewModel side is a simple adjacent swap persisted continuously â€” no "commit on drop" step. Internally each row **must be wrapped in `key(itemKey) { â€¦ }`** â€” without it the non-lazy `Column` identifies children by slot, so the first swap restarts the row's `pointerInput` and cancels the in-progress gesture. `key` must be genuinely unique per row (duplicates of the same exercise are allowed), so rows carry stable ids (`TemplateSlot` a synthetic negative `newId()`; session/set entries already have stable ids).
- **Settings hub** (`ui/settings/SettingsScreen`): a short list of **grouped rows**, not a scroll of inline forms â€” three labelled groups (ERNĂ„HRUNG / BENACHRICHTIGUNGEN / APP) over a version footer, fitting on one screen without scrolling. The rule for what goes where: **more than one control â†’ its own sub-screen** (`Screen.Goals`, `Screen.DataBackup`, `Screen.Support` â€” absent from `bottomNavItems`, so the bar hides itself), **a single control â†’ inline** (the reminder `Switch`; the language row, whose only action is opening its picker sheet). Rows report their current value inline (`goal.kcal`, the picked language) so the common case needs no tap. The hub co-hosts `GoalsViewModel` (for the calorie preview only), `DataViewModel` (reminder toggle) and `LanguageViewModel`; the sub-screens resolve their own via `hiltViewModel()`.
  - **Row vocabulary** (`ui/settings/SettingsRow.kt`): `SettingsGroupLabel` (mono-uppercase caption) + `SettingsGroup` (bordered card) + `SettingsRow` (leading icon, title, optional supporting line, optional `value`, `trailing` slot) + `SettingsRowDivider` (hairline inset past the icon). `onClick` is nullable so a switch row stays non-clickable as a whole. - **Rest timer** (`domain/RestTimer` + `notification/RestTimer*`): checking off a set starts a rest countdown (per-exercise `Exercise.restSeconds`, else 180 s default). Design constraints that shaped it:
  - **Pure state** (`domain/RestTimer`): anchored to a wall-clock `endAtMs`; remaining time is always recomputed from "now" (correct across backgrounding); `start`/`paused`/`resumed`/`adjusted` are pure transitions.
  - **VM stays Android-free**: `WorkoutSessionViewModel` owns the timer state and emits one-shot `RestCommand`s on a channel; the screen mirrors them into the Context-holding `RestTimerService`. There are exactly two: **`Sync`** carries the *complete* timer state (`exerciseName`/`totalSeconds`/`endAtMs`/`pausedRemainingMs`) and is emitted on **every** transition â€” start, resume, pause, Â±15 s â€” so the service never reconstructs anything and an adjustment always moves the real alert instant; **`Stop`** ends the rest (skip/finish/discard). Every transition funnels through the single private `applyRest()` write path (set state â†’ persist anchor â†’ emit `Sync`). There is deliberately **no** "expired" command â€” the service owns the moment the countdown hits zero. The VM's `restTimer` flow is **display-only**: it is bound to the screen's lifecycle, so it must never be responsible for the alert. The in-app `RestTimerBar` (Scaffold `bottomBar`) shows mm:ss + progress with Â±15 s / pause-resume / skip; each exercise-card header carries a rest-duration override chip (persisted via `catalog.upsert` with a new `restSeconds`).
  - **Survives leaving the session**: the anchor is persisted onto the session (`restExerciseStableId`/`restTotalSeconds`/`restEndAtMs`/`restPausedRemainingMs`, null when idle) on every transition; on `init` an active session's anchor is restored (dropped + cleared if already expired).
  - **The service owns the countdown â€” no alarms at all** (`notification/RestTimerService`): a **long-lived foreground service** runs for the *entire* rest and fires the alert itself, on the second. Nothing else can promise that: a countdown in the UI layer dies with the screen's lifecycle (so it never fires while backgrounded â€” that was the old "notification ticks into the negative" bug), and an *inexact* alarm (the only kind this app may use) is deferred by Doze by seconds to minutes. `AlarmManager` is gone from the app entirely. The service loop recomputes the remainder from the **wall clock** every pass and trims the final `delay` to it, so an overshooting tick cannot drift the alert instant; a **partial wake lock** (acquired for `remaining + alert + margin`) keeps the CPU from suspending with the screen off. Verified on an emulator: alert lands at +30.8 s of a 30 s rest with the app foregrounded, backgrounded, **and** with the screen off; two Â±15 s taps move it to +60.6 s; a pause freezes it indefinitely and resume fires the exact remainder.
  - **Still no exact alarms (Play policy)**: `USE_EXACT_ALARM`/`SCHEDULE_EXACT_ALARM` remain deliberately **undeclared** â€” Google Play restricts them to apps whose *core functionality* is an alarm clock/timer/calendar, which a training app is not (Play-Console exact-alarm declaration = "No"). The foreground service delivers the same precision without them.
  - **FGS type `specialUse`** (manifest `android:foregroundServiceType="specialUse"` + `FOREGROUND_SERVICE_SPECIAL_USE` permission + a `PROPERTY_SPECIAL_USE_FGS_SUBTYPE` of `rest_timer_between_sets`): the honest type for a rest-between-sets countdown â€” it is neither media playback nor any other predefined category (the previous `mediaPlayback` declaration was for a service that only played ~2 s of audio). **This needs a justification in the Play Console's foreground-service declaration.** `startForeground` only passes `FOREGROUND_SERVICE_TYPE_SPECIAL_USE` from **API 34** (`startForegroundCompat`) â€” passing a type the platform cannot match to the manifest throws.
  - **Killed with the app**: swiping the app out of the recents kills the service with the process, so that rest's alert is forfeited (an explicitly chosen trade-off â€” self-restarting is unreliable on OEM battery managers). The persisted anchor still restores the timer on the next launch, and `restoreRestTimer` re-runs `applyRest()` rather than only restoring state, so the service is running again even after a process death.
  - **Ongoing notification** (silent LOW channel `rest_timer_ongoing`): custom content view (`notification_rest_timer.xml`) with a large countdown â€” running uses an embedded `Chronometer` counting down to the end instant (Android keeps it live without re-posting), paused shows a frozen mm:ss.   - **Expiry alert** (`RestTimerService.fireAlert`): the service promotes its **own** bundled `res/raw/rest_over.ogg` (~0,9 s, generated with ffmpeg â€” no system ringtone, so the app always sounds like itself) plus a short double-buzz, capped at 2 s (`ALERT_DURATION_MS`). Vibration goes through `Vibrator` directly with alarm usage, never through a notification channel â€” so the channel stays fully silent and nothing ever doubles up. The alert notification is posted via `startForeground` under its own id, which keeps the service a legal FGS while it plays; afterwards `STOP_FOREGROUND_DETACH` + `RestTimerNotifier.repostExpiredDismissible` re-posts it plainly, because a `startForeground` notification carries **`FLAG_NO_CLEAR`** that detaching does not drop (it would otherwise be un-swipeable and clearable only by tapping into the app).
  - **Notification-channel rules** (general): a channel's importance/sound/vibration are **immutable once created** â€” changing any of them requires a **new channel id** (hence `rest_timer_alert_v4`, now fully silent since the service makes the noise itself; `IMPORTANCE_HIGH` is kept so it still pops as a heads-up).
